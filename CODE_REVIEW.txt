Legend:   [BUG]  correctness defect, will misbehave when run

================================================================
0. EXECUTIVE SUMMARY
================================================================

Three things you should fix before doing anything else, because they
change measured behaviour:

  (A) applyBackground.m uses a single `persistent` variable to schedule
      median-background updates across ALL cameras. With N>=2, only
      camera 1's background is ever refreshed; cam 2 (and 3) keep the
      bgMedian computed during their first frame, when the ring buffer
      is mostly zeros. This silently corrupts every multi-camera run.
      See [BUG-1] below. This is the single most important fix.

  (B) Every camera frame is converted to grayscale TWICE — once in
      updateRingBuf (to fill the ring buffer) and again in
      preprocessFrame. At 720p that is ~1.5-3 ms wasted per camera
      per frame, ~5-9% of the 30 fps budget for N=3.

  (C) `clear cams{i}` is used in three files (calibrateExtrinsics,
      validateCalibration, drawSkyMasks) to release webcams. `clear`
      does not accept indexed expressions — it parses the literal
      string "cams{i}". The webcam handles are NOT released, which
      is why a follow-up call (e.g. running validateCalibration after
      calibrateExtrinsics) sometimes fails to open the cameras until
      MATLAB is restarted.

The remaining issues are real but lower-stakes: small performance wins
(~5-15 ms/frame combined), magic numbers leaking outside buildConfig,
half-implemented features whose comments imply they exist, and a few
lines of dead code.

There is nothing in the codebase that needs a redesign. The architecture
described in CLAUDE.md (cell-array-of-N, stateless functions, single
config) is followed in the implemented modules. Most fixes are local.

[BUG-1]  applyBackground.m: shared persistent counter across cameras
---------------------------------------------------------------------
File: applyBackground.m, lines 32-39

    persistent framesSinceUpdate;
    if isempty(framesSinceUpdate), framesSinceUpdate = 0; end
    framesSinceUpdate = framesSinceUpdate + 1;
    if framesSinceUpdate >= 15 || all(bgMedian(:) == 0)
        bgMedian = double(median(ringBuf_i(:,:,1:nFrames), 3));
        framesSinceUpdate = 0;
    end

`persistent` variables are tied to the FUNCTION, not to the caller.
detectBlobs.m calls applyBackground once per camera per frame, in a
loop. So with N=2:

    Frame 1, cam 1 -> counter 0->1, bgMedian is zero -> update, counter -> 0
    Frame 1, cam 2 -> counter 0->1, bgMedian is zero -> update, counter -> 0
    Frame 2, cam 1 -> 1
    Frame 2, cam 2 -> 2
    ...
    Frame 8, cam 1 -> 14->15 -> UPDATE cam1 -> 0
    Frame 8, cam 2 -> 0->1   -> skip
    Frame 16, cam 1 -> 14->15 -> UPDATE cam1 -> 0
    Frame 16, cam 2 -> 0->1   -> skip
    ...

Cam 2's bgMedian is fixed forever at the median of a near-empty ring
buffer (one real frame plus 59 zeros, threshold 35 -> almost every
sky pixel is foreground). The GMM (which IS per-camera, because
fgDetectors{i} is a separate handle object) carries the system, but
the AND with the bad median mask still distorts results.

FIX (recommended): Move the counter into state, indexed by camera.

    % in initSystem.m, after bgMedian/fgDetectors block:
    state.bgFramesSinceUpdate = zeros(1, N);

    % applyBackground.m signature gains framesSinceUpdate, returns it:
    function [mask, bgMedian, framesSinceUpdate] = applyBackground( ...
            frameGray, bgMedian, ringBuf_i, framesSinceUpdate, ...
            fgDetector, cfg)

        framesSinceUpdate = framesSinceUpdate + 1;
        if framesSinceUpdate >= cfg.bgUpdateInterval || all(bgMedian(:) == 0)
            bgMedian = double(median(ringBuf_i(:,:,1:nFrames), 3));
            framesSinceUpdate = 0;
        end
        ...
    end

And update detectBlobs.m / preprocessFrame.m to plumb the counter
through. (This is also the right refactor for [CONV-3] — magic
number 15 — and [CONV-4] — magic number 35 for the threshold.)

NOTE: testSingleCamera.m happens to work correctly only because N=1.
You will not notice this bug until you wire up cam 2.


[BUG-2]  Three callers leak webcam handles ("clear cams{i}")
---------------------------------------------------------------------
calibrateExtrinsics.m line 95:   for i = 1:N, clear cams{i}; end
validateCalibration.m line 63:   for i = 1:N, clear cams{i}; end
drawSkyMasks.m       line 65:    for i = 1:N, clear cams{i}; end

`clear` is a command, it takes literal variable names. `clear cams{i}`
parses as the string "cams{i}" and silently does nothing — the cell
`cams` is not cleared, the underlying webcam objects are not released,
and the camera stays locked.

Symptoms: a second call to `webcam(i)` later in the same MATLAB
session can fail with "Camera could not be opened" or hang. testSingleCamera.m
correctly does `clear cam;` (no braces) and works.

FIX: replace each loop with the cell-clearing form, which DOES release
each handle (delete is implicit when the last reference goes away):

    cams = {};   % drops all references at once
    % or, if you want to keep the cell shape:
    for i = 1:N, cams{i} = []; end

[BUG-5]  Pose-composition convention is not verified
---------------------------------------------------------------------
File: calibrateExtrinsics.m, lines 143-144

    R{i} = R_rel * R{ref};
    t{i} = R_rel * t{ref} + t_rel;

`relativeCameraPose` returns the pose of camera target relative to
camera ref in the convention `X_ref = R_rel * X_target + t_rel` (i.e.
target-to-ref). The composition above is correct only under the
"world-to-camera" interpretation of R, t.

In validateCalibration.m line 91 you then build:
    tform = rigidtform3d(cal.R{i}, cal.t{i}');

`triangulateMultiview` requires the AbsolutePose field to be the
camera's pose in WORLD coordinates (camera-to-world). If R{i}, t{i}
are world-to-camera, you'd need to invert: R_abs = R{i}', t_abs = -R{i}' * t{i}.

This may already be working depending on which MATLAB toolbox version
you have (the convention changed around R2022b when rigidtform3d
replaced rigid3d). Action: run validateCalibration once cameras are
mounted and confirm the reported distance matches your tape-measured
distance to within a few percent. If it's off by exactly the right
ratio but in the wrong sign of one axis, this is the cause.

(I'd not change anything here speculatively — the exact convention
depends on toolbox version and is best verified empirically.)

preprocessFrame.m line 23:  frameGray = rgb2gray(frame);

Either return it from updateRingBuf, or read it back from the ring

    [blobs{i}, state.bgMedian{i}] = preprocessFrame( ...
        frameGray, state.bgMedian{i}, state.ringBuf{i}, ...);

[PERF-2]  median() over 60 frames every 15 frames
---------------------------------------------------------------------
applyBackground.m line 37:
    bgMedian = double(median(ringBuf_i(:,:,1:nFrames), 3));

For 1280x720x60 uint8, MATLAB's median is roughly 80-200 ms on a
laptop CPU — a noticeable spike every half-second. Three options,
in order of effort:

  (a) Use the ring buffer's natural index range. Right now you median
      over slots 1:60 regardless of where the write head is. While
      median is order-independent, when the buffer hasn't filled past
      slot 60 you're including initial zeros. Track a `nFilled` count
      in state and use min(nFilled, cfg.medianBufLen).

  (b) Subsample temporally. Median over every Kth frame in the buffer
      (e.g. 20 frames out of 60). On sky data the median is very
      stable across nearby frames.

  (c) Approximate running median. Maintain bgMedian as
      bgMedian = bgMedian + sign(frameGray - bgMedian) * step;
      with step ~= 1.0/30. This is O(1) per pixel per frame, no
      buffer scan at all. Different statistical properties from a
      true median (slower to react to large changes) but excellent
      for slowly-drifting sky.

(c) is the standard fix for outdoor surveillance. It also makes the
ring buffer's role smaller — at that point you only keep it for
debugging / sky-drift inspection.


[PERF-3]  insertShape and insertText called inside per-blob loop
---------------------------------------------------------------------
renderFrame.m lines 38-45:

    for k = 1:numel(blobs{i})
        c       = blobs{i}(k).centroid;
        display = insertShape(display, 'Circle', [c(1) c(2) 6], ...);
        display = insertText(display, [c(1)+8 c(2)], ...);
    end

Each call rasters and copies the entire HxWx3 image. With 5 blobs you
do 10 image-sized memcpy operations per frame per camera. Both
functions accept arrays of positions in a single call.

FIX:
    centroids = vertcat(blobs{i}.centroid);                % nBlobs x 2
    if ~isempty(centroids)
        circles = [centroids, repmat(6, size(centroids,1), 1)];
        display = insertShape(display, 'Circle', circles, ...
                              'Color', 'green', 'LineWidth', 2);

        areaTexts = arrayfun(@(b) sprintf('%.0fpx2', b.area), blobs{i}, ...
                             'UniformOutput', false);
        textPos   = centroids + [8 0];
        display   = insertText(display, textPos, areaTexts, ...
                               'FontSize', 10, 'BoxOpacity', 0);
    end

This drops a typical 5-blob frame from ~30 ms of rendering to ~5 ms.


[PERF-4]  Dynamic array growth in logFrame and the testSingleCamera benchmarks
---------------------------------------------------------------------
logFrame.m:
    log.timestamps(end+1) = ...
    log.syncFlags(end+1)  = ...
    log.nBlobs(:,end+1)   = ...
    log.blobCentroids{end+1} = ...

testSingleCamera.m:
    t_loop_snapshot(end+1) = ...
    t_loop_detect(end+1)   = ...
    t_loop_display(end+1)  = ...
    t_loop_total(end+1)    = ...
    interFrameTimes(end+1) = ...

Each `(end+1)` reallocates and copies. For a 90 s session at 30 fps
that's 2700 reallocations per field, total cost growing as O(n^2).
Not a real-time killer but it adds tens of milliseconds of GC-style
spikes near the end of long sessions and can show up as bad jitter
numbers.

FIX: preallocate from cfg.

    estFrames = ceil(cfg.fps * nSeconds * 1.2);   % +20% headroom
    log.timestamps  = nan(1, estFrames);
    log.syncFlags   = false(1, estFrames);
    log.nBlobs      = zeros(cfg.N, estFrames);
    log.blobCentroids = cell(1, estFrames);

    % logFrame writes by index, not by end+1:
    function log = logFrame(log, blobs, timestamps, syncOk, frameIdx, cfg)
        log.timestamps(frameIdx)    = mean(timestamps);
        log.syncFlags(frameIdx)     = syncOk;
        log.nBlobs(:,frameIdx)      = cellfun(@numel, blobs).';
        log.blobCentroids{frameIdx} = ...
    end

    % saveSession trims trailing NaNs/empties before saving.

You also need to truncate at session end if the loop exits early.


[PERF-5]  Sequential snapshot() calls
---------------------------------------------------------------------
acquireFrames.m lines 22-25:
    for i = 1:N
        frames{i}     = snapshot(cams{i});
        timestamps(i) = toc(tStart);
    end

snapshot() is blocking. With 2 webcams at ~15-25 ms each you get
30-50 ms total acquisition latency per "frame" — and worse, the
two grabs are serial in time, so the per-frame timestamp jitter
between cam 1 and cam 2 is at least one snapshot's worth (~25 ms),
which already exceeds cfg.maxSyncError = 33 ms minus normal jitter.

There is no easy fix in MATLAB without doing the acquisition in
separate threads (parfeval / a MEX wrapper around DirectShow / OpenCV).
For a 2-camera setup it's borderline. For 3 cameras you will likely
log syncOk=false on a non-trivial fraction of frames.

ACTION: leave it as-is for now and validate empirically. If sync
flags become a problem, the realistic options are:
  - drop to 720p / 30 fps if currently higher;
  - record both cameras' raw frames and align in post (shifts the
    real-time complexity into replaySession);
  - move acquisition to parfeval (requires Parallel Computing Toolbox).

A more nuanced syncCheck would also help — see [STYLE-2].


[PERF-6]  subplot called every frame inside renderFrame
---------------------------------------------------------------------
renderFrame.m line 33: `subplot(1, N, i);` is invoked every frame
per camera. Subplot resolves the layout each call. For N=2-3 the
cost is small (~1-2 ms total) but it's also avoidable.

FIX: cache axes handles in appdata or in a persistent cell, reuse
on subsequent frames. Or build the figure once with `tiledlayout`
and `nexttile`. Optional cleanup; not high priority.

7. NOT-RECOMMENDED CHANGES (deliberately leaving alone)
================================================================

- The flat-vs-subdirectory question is not worth doing twice.
  Pick one direction once.

- Do not rewrite the median-background system to a running median
  yet ([PERF-2c]). The current method is acceptable on a laptop
  for N<=3 at 720p/30 fps. Switch only if profiling shows median
  is the bottleneck, which testSingleCamera will tell you.

- Do not parallelise acquireFrames yet ([PERF-5]). The complexity
  is not justified until empirical sync rates are measured under
  the actual hardware setup.

- The Image Processing Toolbox calls (regionprops, imopen, imclose,
  vision.ForegroundDetector) are correctly chosen and tuned. No
  reason to swap any of these for hand-rolled equivalents.
